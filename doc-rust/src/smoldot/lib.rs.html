18//! Client for Polkadot and Substrate-compatible chains.
19//!
20//! # Overview of a blockchain
21//!
22//! A blockchain is, in its essence, a distributed and decentralized key-value database. The
23//! principle of a blockchain is to make it possible for any participant to perform modifications
24//! to this database, and for all participants to eventually agree on the current state of said
25//! database.
26//!
27//! In Polkadot and Substrate-compatible chains, the state of this database is referred to as
28//! "the storage". The storage can be seen more or less as a very large `HashMap`.
29//!
30//! A blockchain therefore consists in three main things:
31//!
32//! - The initial state of the storage at the moment when the blockchain starts.
33//! - A list of blocks, where each block represents a group of modifications performed to the
34//! storage.
35//! - A peer-to-peer network of clients connected to each other and exchanging information such
36//! as newly-produced blocks.
37//!
38//! Blocks are built on top of each other, forming a sequential list of modifications to the
39//! storage on top of its initial state.
40//!
41//! ## Blocks
42//!
43//! A block primarily consists in three properties:
44//!
45//! - A parent block, referred to by its hash.
46//! - An ordered list of **extrinsics**, representing a state change about the storage. An
47//! extrinsic can be either a **transaction** or an **intrinsic**.
48//! - A list of **digest log items**, which include information necessary to verify the
49//! authenticity of the block, such as a cryptographic signature of the block made by its author.
50//!
51//! In order to make abstractions easier, there also exists what is called the **genesis block**,
52//! or block number 0. It doesn't have any parent, extrinsic, or digest item.
53//!
54//! From these three block properties, the following other properties can be derived:
55//!
56//! - The **hash** of the block. This is a unique 32 bytes identifier obtained by hashing all the
57//! block's information together in a specific way.
58//! - The **block number**. It is equal to the parent's block number plus one, or equal to zero
59//! for the genesis block
60//! - The state of the storage. It consists in the state of the storage of the parent block, on
61//! top of which the block's extrinsics have been applied. The state of the storage of the genesis
62//! block is the initial state.
63//!
64//! > **Note**: Not all these properties are held in memory or even on disk. For instance, the
65//! >           state of the storage is quite large and holding a full version of the storage of
66//! >           every single block in the chain would be unrealistic.
67//!
68//! ## Trie
69//!
70//! The **trie** is a data structure that plays an important role in the way a blockchain
71//! functions.
72//!
73//! It consists in a tree of nodes associated with a key, some of these nodes containing a value.
74//! Each node is associated with a hash called the **Merkle value** that encompasses the Merkle
75//! values of the node's children. The Merkle value of the root node of the tree is designated
76//! as "the Merkle trie root" or "the trie root".
77//!
78//! See the [`trie`] module for more details.
79//!
80//! ## Block headers
81//!
82//! In practical terms, when a block needs to be stored or transferred between machines, it
83//! is split in two parts: a **header** and a **body**. The body of a block simply consists in its
84//! list of extrinsics.
85//!
86//! > **Note**: The body of a block and the list of extrinsics of a block are the exact same
87//! >           thing, and these two designations can be used interchangeably.
88//!
89//! A block's header contains the following:
90//!
91//! - The hash of the parent of the block.
92//! - The block number.
93//! - The **state trie root**, which consists in the trie root of all the keys and values of the
94//! storage of this block.
95//! - The **extrinsics trie root**, which consists in the Merkle root of a trie containing the
96//! extrinsics of the block.
97//! - The list of digest log items.
98//!
99//! The hash of a block consists in the `blake2` hash of its header.
100//!
101//! See the [`header`] module for more information.
102//!
103//! ## Runtime
104//!
105//! The state of the storage of each block is mostly opaque from the client's perspective. There
106//! exists, however, a few hard coded keys, the most important one being
107//! `[0x3a, 0x63, 0x6f, 0x64, 0x65]` (which is the ASCII encoding of the string `:code`). The
108//! value associated with this key must always be a [WebAssembly](https://webassembly.org/) binary
109//! code called the **runtime**.
110//!
111//! This WebAssembly binary code must obey a certain ABI and is responsible for the following
112//! actions:
113//!
114//! - Verifying whether a block has been correctly created by its author.
115//! - Providing the list of modifications to the storage performed by a block, given its header
116//! and body.
117//! - Generating the header of newly-created blocks.
118//! - Validating transactions received from third-parties, in order to later be potentially
119//! included in a block.
120//! - Providing the tools necessary to create transactions.
121//!
122//! In other words, the runtime is responsible for all the actual *logic* behind the chain being
123//! run. For instance, when performing a transfer of tokens between one account and another, it is
124//! the runtime WebAssembly code that verifies that the balances are correct and updates them.
125//!
126//! Since applying an extrinsic modifies the state of the storage, it is possible for an
127//! extrinsic to modify this WebAssembly binary code. This is called a **runtime upgrade**. In
128//! other words, any block can potentially modify the logic of the chain.
129//!
130//! See also the [`executor`] module for more information.
131//!
132//! ## Forks and finalization
133//!
134//! While blocks are built on top of each other, they don't necessarily form one single chain. It
135//! is possible for multiple different blocks to have the same parent, thereby forming multiple
136//! chains. This is called a **fork**.
137//!
138//! When multiple chains fly around in the network, each node select one chain they consider as
139//! the "best". However, because of latency, disconnects, or similar reasons, it is possible for
140//! a node to not be immediately aware of the existence of a chain that it would consider as a
141//! better chain than its current best. Later, when this node learns about this better chain, it
142//! will need to perform a **reorg** (short for *reorganization*), where blocks that were part of
143//! the best chain no longer are.
144//!
145//! In order to avoid troublesome situations arising from reorgs, Substrate/Polkadot provides the
146//! concept of **finality**. Once a block has been finalized, it is guaranteed to always be part
147//! of the best chain. By extension, the parent of a finalized block is always finalized as well.
148//! The genesis block of a chain is by definition is always finalized.
149//!
150//! As such, a chain of blocks is composed of two parts: one **finalized** part (usually the
151//! longest), and one non-finalized part. While the finalized part is a single linear list of
152//! blocks, the non-finalized part consists in a *directed rooted tree*.
153//!
154//! In order to finalize a block, Substrate/Polkadot nodes use the **GrandPa** algorithm. Nodes
155//! that are authorized to do so by the runtime emit votes on the peer-to-peer network. When two
156//! thirds or more of the authorized nodes have voted for a specific block to be finalized, it
157//! effectively becomes finalized.
158//!
159//! After these votes have been collected, they are collected in what is called a
160//! **justification**. This justification can later be requested by nodes who might not have
161//! received all the votes, or for example if they were offline.
162//!
163//! See the [`chain`] and [`finality`] modules for more information.
164//!
165//! # Usage
166//!
167//! This library intentionally doesn't provide any ready-to-use blockchain client. Instead, it
168//! provides tools that can be combined together in order to create a client.
169//!
170//! Components that are most certainly needed are:
171//!
172//! - Connectivity to a peer-to-peer network. See the [`network`] module.
173//! - A persistent storage for the blocks. See the [`database`] module.
174//! - A state machine that holds information about the state of the chain and verifies the
175//! authenticity and/or correctness of blocks received from the network. See the [`sync`]
176//! module.
177//!
178//! Optionally:
179//!
180//! - The capacity to author new blocks. This isn't implemented as of the writing of this
181//! documentation.
182//! - A JSON-RPC client, in order to put a convenient-to-use UI on top of the client. See the
183//! [`json_rpc`] module.
184//! - TODO: telemetry
185//!
