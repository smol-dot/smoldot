18//! Syncing, short for synchronizing, consists in synchronizing the state of a local chain with
19//! the state of the chain contained in other machines, called *remotes* or *sources*.
20//!
21//! > **Note**: While the above summary is a bit abstract, in practice it is almost always
22//! >           done by exchanging messages through a peer-to-peer network.
23//!
24//! There exists multiple syncing strategies, depending on the amount of information that is.
25//! desired and the distance between the highest block of the local chain and the highest block
26//! available from the syncing sources.
27//! The [`all`] module provides a state machine that combines several syncing strategies into one.
28//!
29//! # Security considerations
30//!
31//! ## Eclipse attacks
32//!
33//! While there exists various trade-offs between syncing strategies, safety is never part of
34//! these trade-offs. All syncing strategies are safe, in the sense that malicious remotes cannot
35//! corrupt the state of the local chain.
36//!
37//! It is possible, however, for a malicious remote to omit some information, such as the
38//! existence of a specific block. If all the remotes that are being synchronized from are
39//! malicious and collude to omit the same information, there is no way for the local node to
40//! learn this information or even to be aware that it is missing an information. This is called
41//! an **eclipse attack**.
42//!
43//! For this reason, it is important to ensure a large number and a good distribution of the
44//! sources. In the context of a peer-to-peer network where machines are picked randomly, a
45//! minimum threshold of around 7 peers is generally considered acceptable. Similarly, in the
46//! context of a peer-to-peer network, it is important to establish outgoing connections to other
47//! nodes and not only rely on incoming connections, as there is otherwise the possibility of a
48//! single actor controlling all said incoming connections.
49//!
50//! ## Malicious runtimes
51//!
52//! Synchronizing the chain requires executing a piece of WebAssembly code known as the runtime.
53//! Each chain has a different runtime, stored on chain, and this runtime can be modified as part
54//! of a the transactions in a block.
55//!
56//! There are two safety considerations concerning executing runtime code:
57//!
58//! - The execution could take a lot of time, or possibly be stuck in an infinite loop.
59//! - The execution could modify a lot of storage items. Smoldot needs to store every single
60//! storage item change, and a lot of modification could make the memory usage of smoldot explode.
61//!
62//! The Substrate model, and thus the smoldot implementation, intentionally do not do anything to
63//! address these two safety issues. In other words, the runtime of a chain is considered to never
64//! be malicious. After all, a chain whose runtime is intentionally trying to crash a client is
65//! borked and there is no reason to connect to it.
66//!
