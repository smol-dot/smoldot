ยงallow_unknown_consensus_engines: bool

If false, blocks containing digest items with an unknown consensus engine will fail to verify.

Note that blocks must always contain digest items that are relevant to the current consensus algorithm. This option controls what happens when blocks contain additional digest items that arenโt recognized by the implementation.

Passing true can lead to blocks being considered as valid when they shouldnโt, as these additional digest items could have some logic attached to them that restricts which blocks are valid and which are not.

However, since a recognized consensus engine must always be present, both true and false guarantee that the number of authorable blocks over the network is bounded.

Pre-allocated capacity for the number of block sources.

ยงblocks_capacity: usize

Pre-allocated capacity for the number of blocks between the finalized block and the head of the chain.

Should be set to the maximum number of block between two consecutive justifications.

Maximum number of blocks of unknown ancestry to keep in memory.

Maximum number of simultaneous pending requests made towards the same block.

ยงdownload_ahead_blocks: NonZero<u32>

Number of blocks to download ahead of the best verified block.

Whenever the latest best block is updated, the state machine will start block requests for the block best_block_height + download_ahead_blocks and all its ancestors. Considering that requesting blocks has some latency, downloading blocks ahead of time ensures that verification isnโt blocked waiting for a request to be finished.

The ideal value here depends on the speed of blocks verification speed and latency of block requests.

ยงdownload_bodies: bool

If true, the body of a block is downloaded (if necessary) before a ProcessOne::VerifyBlock is generated.

ยงdownload_all_chain_information_storage_proofs: bool

If true, all the storage proofs and call proofs necessary in order to compute the chain information of the warp synced block will be downloaded during the warp syncing process. If false, the finality information of the warp synced block is inferred from the warp sync fragments instead.

ยงcode_trie_node_hint: Option<ConfigCodeTrieNodeHint>

Known valid Merkle value and storage value combination for the :code key.

If provided, the warp syncing algorithm will first fetch the Merkle value of :code, and if it matches the Merkle value provided in the hint, use the storage value in the hint instead of downloading it. If the hint doesnโt match, an extra round-trip will be needed, but if the hint matches it saves a big download.
